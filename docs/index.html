A manifesto, and a working demo

Why R.A.G is Dead

Retrieval was never supposed to mean stuffing a wall of chunks into a context window and praying. It should mean verified microagents fetching only what is true, and proving where every word came from.

Watch it work Read the source

RAG asks you to trust a black box

Classic retrieval-augmented generation embeds your data, pulls the top-k nearest chunks, and dumps them into one giant prompt. Three things break:

  • Hallucinated relevance. Cosine similarity is not understanding. The "nearest" chunk is often the wrong one, and the model cannot tell.
  • No provenance. The answer cites nothing you can check. You are trusting that the chunk said what the model claims it said.
  • Context bloat. To be safe, RAG over-fetches. Tens of thousands of tokens of maybe-relevant text, every single call.

How it actually works

  1. Planner (Claude, ~500 tokens, never sees the codebase) decomposes the query into 3 to 5 atomic shell tasks scoped to the right directories.
  2. Workers (local Ollama qwen2.5:1.5b, parallel, fresh context each) run sandboxed grep/sed and return structured citations.
  3. Manager verifies every citation with deterministic shell (sed -n '{line}p'). No model checks another model.
  4. Main LLM (Claude) sees only the verified context and answers with real file:line citations.

The thing RAG structurally cannot do: prove the retrieved fact is real before answering.

Every claim has a receipt

NEXUS does not ask you to trust a chunk. Each statement in the answer links to a verified file:line that a shell command confirmed exists. Provenance is the product, not an afterthought.

Retrieval is not dead. Unverified retrieval is.
