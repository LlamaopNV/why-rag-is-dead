Why R.A.G is Dead

Stop stuffing chunks into a context window and praying. Verified microagents fetch only what is true.

Watch it work Read the source

RAG asks you to trust a black box

Retrieval was never supposed to mean stuffing a wall of chunks into a context window and praying. Classic retrieval-augmented generation embeds your data, pulls the top-k nearest chunks, and dumps them into one giant prompt. Three things break.

  1. 01

    Hallucinated relevance

    Cosine similarity is not understanding. The "nearest" chunk is often the wrong one, and the model cannot tell.

  2. 02

    No provenance

    The answer cites nothing you can check. You are trusting that the chunk said what the model claims it said.

  3. 03

    Context bloat

    To be safe, RAG over-fetches. Tens of thousands of tokens of maybe-relevant text, every single call.

How it actually works

  1. Planner

    Claude, about 500 tokens, never sees the codebase. It decomposes the query into 3 to 5 atomic shell tasks scoped to the right directories.

  2. Workers

    Local Ollama qwen2.5:1.5b, parallel, fresh context each. They run sandboxed grep and sed and return structured citations.

  3. Manager

    Verifies every citation with deterministic shell, sed -n '{line}p'. No model checks another model.

  4. Main LLM

    Claude sees only the verified context and answers with real file:line citations.

The thing RAG structurally cannot do: prove the retrieved fact is real before it answers.

NEXUS does not ask you to trust a chunk. Every statement links to a verified file:line that a shell command confirmed exists.

Provenance is the product, not an afterthought.
